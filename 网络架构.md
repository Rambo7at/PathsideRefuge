## 二、核心类职责

### 1. NetCore (Autoload)
- 连接管理（ENet 主机/客户端、LAN 发现）
- 自定义 RPC 发送/接收（SendRpcToPeer、BroadcastRpc、ReceiveCustomPackets）
- 分布式对象存储（Dictionary<NetID, NetObj>）
- 同步调度（20Hz 脏数据检测与发送）
- 所有权判断与修订号维护

### 2. NetObjManager (Node)
- 监听 NetCore.OnSpawned / OnDestroyed
- 持有预制体哈希映射 Dictionary<int, PackedScene>
- 实例化/销毁网络对象节点，并调用 NetSyncBase.Setup() 绑定 NetObj

### 3. NetSyncBase (Node)
- 绑定 NetObj，提供 IsOwner、NetID
- 注册对象 RPC 方法，发送 InvokeRpc / InvokeRpcToAll
- **新增功能**：自我网络化（详见第三章）

### 4. NetTransformSync (Node)
- Owner 端：定时写入 Position/Rotation 至 NetObj 并递增 DataRevision
- 非 Owner 端：从 NetObj 读取目标位置并做 Lerp/Slerp 平滑插值

### 5. ItemManager
- 管理所有“可拾取物品”的预制体（武器、药水等）
- 提供 HasPrefab(string path) 供 NetSyncBase 查询物品是否合法
- 与 NetObjManager 无直接依赖，各自独立注册

## 三、静态场景物品网络化方案

### 问题
手工拖入场景的功能性物品（如可拾取物）未经过 `Spawn` 流程，`NetCore` 中无对应 `NetObj`，无法被同步。

### 解决思路
利用 `NetSyncBase` 在初始化时的自主判断能力，如果发现自身没有被注入 `NetObj`（即不是由 `Spawn` 生成），且当前在主机端，则主动向 `ItemManager` 查询自身预制体路径是否已注册。如果已注册，则调用 `NetCore.Spawn` 为自己生成网络身份并销毁本地占位节点。

### 具体流程
1. `NetSyncBase._Ready()`
   - 若 `NetObj != null` → 标准模式（由 Spawn 产生），正常初始化
   - 否则：
     - 若不是主机 → 作为客户端占位节点，隐藏或自身销毁，等待网络对象到来
     - 若是主机：
       - 检查 `ItemManager.HasPrefab(PrefabPath)`
         - 是 → `NetCore.Spawn(PrefabPath, GlobalPosition, GlobalRotation)` → `QueueFree()` 销毁本地占位
         - 否 → 视为静态装饰，保持原样

### 优点
- 零场景侵入：不需额外挂载 ServerSpawner 或添加节点组
- 预制体自包含：任何场景拖入该预制体都能自动网络化
- 完美兼容后期动态生成

### 前置条件
- `NetSyncBase` 上需暴露 `[Export] string PrefabPath` 用来记录自身资源路径
- `ItemManager` 提供 `HasPrefab(string path)` 方法
- 确保 `NetCore` 和 `ItemManager` 在 `NetSyncBase._Ready` 之前已完成初始化

## 五、开发优先级

1. **完善接收端与测试闭环**：在 NetCore._Process 中调用 ReceiveCustomPackets，验证 ObjCreate/ObjSync 通路。
2. **实现 NetSyncBase 主动模式**，并配合 ItemManager 使手动放置的物品自动网络化。
3. **重构 PlayerManager**，使玩家通过 NetObjManager 生成。
4. **集成测试**：手工放置物品和动态生成物品均能在客户端正确显示和同步。

> 以上设计确保项目从静态手工搭建地图平滑过渡到未来的随机地图生成，所有网络对象统一由 NetCore 调度。
